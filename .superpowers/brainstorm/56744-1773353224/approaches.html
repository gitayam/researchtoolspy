Architecture: Which approach for the Cross Table engine?

Three options for how the matrix, scoring, and analysis engine is structured

A

Monolithic Page Component

Single CrossTablePage with all logic inline — matrix rendering, scoring, weighting, AI, export. Similar to how ACH is built today.

Pros

  • Fastest to build — one file, one mental model
  • Matches existing ACH pattern
  • Simple state management (useState)

Cons

  • Will grow large fast (8 templates + 6 scoring methods + AI + collab + export)
  • Hard to test individual features
  • Template logic mixed with scoring logic mixed with UI
B

Composable Engine + UI Layer

Separate the cross-table engine (scoring, weighting, AHP, sensitivity) from the UI layer (matrix grid, template forms, visualization). Engine is a pure TypeScript module, UI consumes it via a context provider.

Pros

  • Engine is testable without UI — unit test scoring, AHP, sensitivity
  • Templates are config-driven (add new template = add config object)
  • UI components stay focused (Grid, ScoreCell, WeightSlider, ResultsChart)
  • Collaborative scoring is a layer on top, not mixed in

Cons

  • More files upfront (~12-15 vs ~4-5)
  • Slightly more abstraction to navigate

⭐ Recommended — The feature set (8 templates × 6 scoring methods × AI × collab × export) demands separation. The engine handles the math, the UI handles the pixels.

C

Plugin Architecture

Each template, scoring method, and weighting algorithm is a self-contained plugin with a standard interface. A registry discovers and composes them at runtime.

Pros

  • Maximum extensibility — add templates/scoring without touching core
  • Could allow user-defined scoring plugins in the future

Cons

  • Over-engineered for current scope — YAGNI
  • Plugin interfaces add complexity before there's a clear need
  • Slower to build, harder to debug
